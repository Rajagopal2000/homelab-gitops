# Homelab App-of-Apps Design

## Overview

ArgoCD App-of-Apps setup for a single-node Kubernetes homelab cluster running on Proxmox VMs. Uses ApplicationSets with git directory generators to auto-discover applications across three layers with sync wave ordering.

## Infrastructure

- **Cluster:** Single-node Kubernetes on Proxmox VM
- **GitOps:** ArgoCD (Helm-installed, self-managed after bootstrap)
- **Secrets:** Bitnami Sealed Secrets
- **Certificates:** cert-manager with Let's Encrypt via Cloudflare DNS-01 challenge
- **Storage:** Longhorn
- **Ingress:** Traefik (LAN access) + Cloudflare Tunnel (external access)
- **Domain:** Cloudflare-registered domain

## Application Pattern

Every application follows the same thin Helm wrapper pattern:

### Chart.yaml

Declares the upstream Helm chart as a dependency:

```yaml
apiVersion: v2
name: <app-name>
version: 0.1.0
dependencies:
  - name: <upstream-chart-name>
    version: "<version>"
    repository: "<upstream-helm-repo-url>"
```

### values.yaml

Overrides namespaced under the dependency name:

```yaml
<upstream-chart-name>:
  key: value
```

### templates/ (optional)

For SealedSecret manifests or other custom resources the app needs.

### Adding a new app

1. Create a directory under the correct layer (`infra/`, `platform/`, or `applications/`)
2. Add `Chart.yaml` with the upstream chart dependency
3. Add `values.yaml` with overrides
4. Commit and push — the ApplicationSet auto-discovers it

## ApplicationSet Design

One ApplicationSet per layer. Each uses a git directory generator to auto-discover app directories.

### Template (example: infra)

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: infra
  namespace: argocd
  annotations:
    argocd.argoproj.io/sync-wave: "0"
spec:
  goTemplate: true
  generators:
    - git:
        repoURL: <repo-url>
        revision: HEAD
        directories:
          - path: infra/*
  template:
    metadata:
      name: "infra-{{.path.basename}}"
      namespace: argocd
      labels:
        app.kubernetes.io/part-of: homelab
        app.kubernetes.io/component: infra
    spec:
      project: default
      source:
        repoURL: <repo-url>
        targetRevision: HEAD
        path: "{{.path.path}}"
        helm:
          valueFiles:
            - values.yaml
      destination:
        server: https://kubernetes.default.svc
        namespace: "{{.path.basename}}"
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
        syncOptions:
          - CreateNamespace=true
```

### Layer ordering via sync waves

| Wave | Layer        | ApplicationSet | Apps                                      |
|------|-------------|----------------|-------------------------------------------|
| 0    | infra       | infra.yaml     | sealed-secrets, cert-manager, longhorn     |
| 1    | platform    | platform.yaml  | traefik, cloudflare-tunnel                 |
| 2    | applications| applications.yaml | future workloads                       |

### Intra-layer ordering

Applications generated by the same ApplicationSet sync independently and in parallel. There is no strict ordering within a layer. For the infra layer, sealed-secrets and cert-manager will both start deploying simultaneously. The cert-manager SealedSecret (Cloudflare API token) will fail to decrypt until the sealed-secrets controller is running — ArgoCD's automated retry and selfHeal will resolve this naturally once sealed-secrets is healthy. No manual intervention needed; the system converges on its own.

## Sync Policy

- **Automated sync:** enabled on all ApplicationSets
- **Prune:** enabled — resources removed from git are deleted from the cluster
- **Self-heal:** enabled — manual drift is corrected automatically
- **CreateNamespace:** enabled — each app gets its own namespace matching the directory name

## Namespace Strategy

Each application deploys to a namespace matching its directory name:

| App              | Namespace          |
|------------------|--------------------|
| sealed-secrets   | sealed-secrets     |
| cert-manager     | cert-manager       |
| longhorn         | longhorn           |
| traefik          | traefik            |
| cloudflare-tunnel| cloudflare-tunnel  |

## Secrets Management

Secrets are managed via Bitnami Sealed Secrets:

1. Create a regular Kubernetes Secret locally
2. Encrypt with `kubeseal` CLI against the cluster's sealed-secrets controller
3. Commit the resulting SealedSecret manifest to `templates/` in the relevant app directory

Known SealedSecrets needed:
- `infra/cert-manager/templates/cloudflare-api-token.yaml` — Cloudflare API token for DNS-01 challenge
- `platform/cloudflare-tunnel/templates/tunnel-credentials.yaml` — Cloudflare Tunnel credentials

## Bootstrap Process

### Prerequisites

- Kubernetes cluster running and `kubectl` configured
- Helm installed locally
- Git remote (GitHub, Gitea, etc.) accessible from the cluster

### One-time manual steps

```bash
# 1. Add ArgoCD Helm repo
helm repo add argo https://argoproj.github.io/argo-helm
helm repo update

# 2. Install ArgoCD
helm install argocd argo/argo-cd \
  -n argocd --create-namespace \
  -f bootstrap/argocd/values.yaml

# 3. Apply the root Application
kubectl apply -f bootstrap/root.yaml
```

After step 3, ArgoCD takes over:
- Root Application syncs `appsets/` directory
- Each ApplicationSet discovers and deploys its layer's apps
- ArgoCD manages its own upgrades via `bootstrap/argocd/`

### Post-bootstrap

- Seal and commit required secrets (Cloudflare API token, tunnel credentials)
- Verify all apps are synced: `kubectl get applications -n argocd`

## ArgoCD Self-Management

ArgoCD is self-managed after bootstrap:
- `bootstrap/argocd/Chart.yaml` wraps the upstream `argo/argo-cd` Helm chart
- `bootstrap/argocd/values.yaml` contains ArgoCD configuration
- The root Application includes ArgoCD itself, so config changes in git auto-apply

## Design Decisions

1. **Helm everywhere** — consistency across all apps; no mix of Helm and Kustomize
2. **ApplicationSets over individual Application manifests** — less boilerplate; adding an app is just creating a directory
3. **Namespace per app** — clean isolation; namespace name matches directory name
4. **Sync waves for layer ordering** — infra deploys before platform deploys before applications
5. **Auto-sync with prune and selfHeal** — full GitOps; the repo is the source of truth
6. **Thin wrapper charts** — each app is a Chart.yaml (upstream dependency) + values.yaml (overrides), not a forked chart
